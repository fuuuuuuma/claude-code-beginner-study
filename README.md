## 1. そもそも Claude Code とは

### 1-1. 普通のChatGPT・Claudeとの違い

普段使っている Claude や ChatGPT は **「聞いたら答える」** チャット型AIです。
**Claude Code は、あなたのPCの中で動いて、ファイルを読み・書き・コマンドを実行できるAI** です。

| 普通のチャットAI | Claude Code |
| --- | --- |
| 文章で答えるだけ | ファイルを読み・書き・コマンド実行 |
| あなたがコピペで橋渡し | あなたのPC上で完結 |
| 毎回ゼロから説明 | プロジェクトのルールを覚えてくれる |
| 1人で何でもやらされる | 専門家チームに分業させられる |

### 1-2. 動画編集者にとっての「効く」ポイント

Claude Code は動画制作のワークフローと相性が抜群です。

- Premiere Pro の XML を読んで、無音カット・ジェットカットを自動化
- Whisper で文字起こし → SRT 字幕生成 → 改行整形までを自動チェーン
- 動画から note 記事 / X投稿文 / YouTube 概要欄まで自動下書き
- 「毎回同じ前置き」を書かなくて済むようにルールを永続化できる

### 1-3. インストールと起動（5分）

```bash
# 標準：npm でグローバルインストール
npm install -g @anthropic-ai/claude-code

# native install（提供状況・対応OSは公式ページで要確認）
curl -fsSL https://claude.ai/install.sh | bash

# プロジェクトのフォルダに移動して起動
cd ~/your-project
claude
```

> ⚠️ インストール手順や提供チャネルは更新されることがあります。**最新の標準手順は必ず[公式ドキュメント](https://docs.anthropic.com/en/docs/claude-code) で確認**してください。
> この資料では初心者向けに **npm / native install** の例を載せています。**Homebrew を使う場合** は、公式ドキュメントの **cask コマンド（`brew install --cask claude-code` 形式）** と、更新方法（`brew upgrade --cask` など）を必ず確認してください。

初回起動でログインを求められたら、ブラウザでサインインするだけ。

### 1-4. まず覚える基本コマンド（最低限）

| コマンド | 何をする |
| --- | --- |
| `claude` | 起動。今いるフォルダがプロジェクトになる |
| `/init` | プロジェクト用の `CLAUDE.md` のたたき台を自動生成 |
| `/cost` | 今のセッションでいくら使ったか確認 |
| `/clear` | 会話履歴をリセット（コンテキスト掃除） |
| `/memory` | プロジェクトの記憶（CLAUDE.md）を編集 |
| `/help` | コマンド一覧 |

> 💡 **最初の一手は `/init`**。これだけでプロジェクトに合った `CLAUDE.md` がたたき台として作られます。

## 2. 最初に押さえる3つの登場人物

Claude Code を理解するうえで、まず **「3つの登場人物」** を区別できれば9割勝ちです。

| 登場人物 | 役割 | ざっくり例え |
| --- | --- | --- |
| **CLAUDE.md** | プロジェクトのルール書 | 会社の就業規則 |
| **skills/** | 自動で発動する手順書 | バイト用マニュアル |
| **agents/** | 専門家として呼ばれる別AI | 外注の専門家 |

### 違いを1行で

- **CLAUDE.md** = 「うちのチームではこう動いてね」という **常駐ルール**
- **skills/** = 「議事録を頼まれたらこの手順で動いてね」という **発動型マニュアル**
- **agents/** = 「調査だけは別のAIに任せたい」という **分業先**

この3つを使い分けるのが、Claude Code 設計の基本です。

## 3. `.claude/` フォルダ全体像（基礎）

```
your-project/
├── CLAUDE.md             ← セッション開始時に必ず読まれる全体ルール
├── CLAUDE.local.md       ← 個人設定（チーム共有しない／.gitignoreする）
└── .claude/
    ├── settings.json     ← ツール権限のallow/deny
    ├── commands/         ← /xxx で明示的に呼ぶカスタムコマンド
    ├── skills/           ← 会話内容から自律発動するワークフロー
    └── agents/           ← 専門分業のサブエージェント
```

### なぜ最初に「構造」を理解するのか

- どこを編集すれば狙った挙動になるかが分かる
- 設定ミスを早い段階で防げる
- 後からの改善が **追記・差分更新** で済むようになる

> 📝 **覚え方**：CLAUDE.md は「常時ON」、skills/agents は「呼ばれたら動く」。

## 4. `CLAUDE.md` の書き方 — AIの「就業規則」

### 4-1. CLAUDE.md とは何か

セッション開始時に **必ず** 読み込まれるファイル。
ここに書いた内容は、そのプロジェクト中ずっと AI の頭の中に居続けます。

### 4-2. 鉄則は「短く・具体的に・トリガー＋アクション」

❌ ダメな例（長くて抽象的）：

```
丁寧に質問に答えてください。コードは綺麗にしてください。エラーには注意してください。
```

⭕ 良い例（トリガー＋アクション）：

```
- レビューを依頼されたら、重大リスクから先に列挙すること
- 外部送信の前に、必ず実行前確認を行うこと
- タスク更新時は原本ファイルのみ更新すること
```

### 4-3. 削除テスト

「**この行を消したら何か困ることが起きる？**」を毎行に問う。
答えが「特に困らない」なら、その行は不要。

### 4-4. 200行以内に収める

CLAUDE.md は毎セッション読み込まれます。
**長い ＝ 毎回トークンを払う ＝ 出力もブレる**。コア原則に絞る。

### 4-5. よくあるQ&A

> **Q. 最初から完璧に作る必要は？**
> A. 不要。最小構成で始め、運用しながら追記するのが実用的。

> **Q. CLAUDE.md は長いほど良い？**
> A. 逆。短く明確な方が効果的。詳細は `skills/` 配下に逃がす（後述）。

## 5. `settings.json` — 安全設計の入り口

`.claude/settings.json` で、AI が使えるツールを制御できます。

```json
{
  "permissions": {
    "allow": ["Bash(git status)", "Bash(npm test)"],
    "deny":  ["Read(./.env)", "Bash(rm -rf *)"]
  }
}
```

| 設定 | 意味 |
| --- | --- |
| `allow` | 確認なしで実行 |
| `deny` | 実行を禁止（最優先） |
| 未指定 | 実行前にユーザー確認 |

### `.env` deny を入れる意味

- APIキーなど機密情報の誤読防止
- 外部共有時の漏洩リスク低減
- 初回で安全設計を固定できる

> 💡 **最初に書くべき1行**：`"deny": ["Read(./.env)"]`

## 6. スキルを作る・育てる（旧 第2回）

### 6-1. そもそもスキルとは？

**スキル＝「特定の依頼が来たら、AIが自動で発動する手順書」**

例：
- 「議事録を作って」と言うと → `skills/議事録作成/` が自動起動
- 「字幕作って」と言うと → `skills/字幕生成/` が自動起動

あなたが毎回手順を説明する必要がなくなります。

### 6-2. 最小のスキルファイル

```
.claude/skills/字幕生成/
└── SKILL.md
```

```markdown
---
name: 字幕生成
description: WAV音声と動画の文字起こしから、Premiere Pro用のSRT字幕を生成する
---

## 手順
1. 音声ファイルを Whisper で文字起こし
2. 句読点で改行を整える（ただし日本語の文脈を見て調整）
3. SRT形式で `output/` に書き出す

## Gotchas（後述）
- まだ空でOK。使いながら埋めていく
```

`description` に書いた内容を頼りに、Claude が「これは字幕生成スキルの仕事だ」と判断して自動発動します。

### 6-3. 🌟 Gotchas — 使うほど賢くなるためのキモ

スキルを使っていると、AI が **何度も同じパターンで間違える** ことがあります。
それを記録しておいて、次回から先回りで伝えるのが Gotchas です。

> Claude Code 利用者・開発者の記事やコミュニティで「スキル設計の中で最も価値が高い」と繰り返し挙げられている実践知です（Anthropic 公式の必須項目ではありません）。

| 通常の指示 | Gotchas |
| --- | --- |
| 「こうしてください」（理想の動作） | 「ここで失敗しやすいので注意」（実際の失敗） |
| 最初から書ける | 使いながら育てる |

**育て方は超シンプル**：

```
スキル実行 → AIがミス → 「なぜミスしたか」を1行追記 → 次回から出なくなる
```

例（字幕生成スキル）：

```markdown
## Gotchas
- 5000文字超だと改行判断が荒くなる → 1000文字ずつ分割して処理
- 「えーと」「あのー」のフィラーを残してしまう → 削除指示を明示
- SRTのタイムコードがズレる → ffprobe で実尺を必ず再計算
```

> 💡 **これが「使うほど賢くなるAI」の本体です**。

### 6-4. Progressive disclosure — フォルダで分割設計

`SKILL.md` に全部書くと、毎回ぜんぶ読み込まれて **重い・高い・精度が落ちる**。
必要なときだけ詳細を読ませるのが Progressive disclosure です。

```
.claude/skills/字幕生成/
├── SKILL.md              ← 概要・トリガー・基本手順だけ（毎回読まれる）
├── references/
│   ├── format_rules.md   ← 詳細フォーマット（必要時のみ参照）
│   └── examples.md       ← 過去の良い字幕例
├── assets/
│   └── template.srt      ← テンプレート
└── data/
    └── log.json          ← 実行ログ（永続データ）
```

`SKILL.md` には一言だけ書けばOK：

```
詳細フォーマットは references/format_rules.md を参照。
過去の良い例は references/examples.md を参照。
```

### 6-5. データ永続化 — スキルが「過去」を覚える

通常スキルは毎回リセット。`data/log.json` に追記指示を書くだけで「先週との差分」が分かるようになります。

```markdown
## データ保存
- 毎回の実行結果を data/log.json に追記すること
- 次回実行時は data/log.json を読んで前回との差分を検出すること
```

向いている業務：
- 毎朝のタスク通知 → 昨日 vs 今日の比較
- 週次レポート → 過去4週分のトレンド分析
- 数値管理 → 先週比・先月比の自動計算

### 6-6. スキル連携チェーン — スキルAの結果をスキルBへ

`description` に「○○スキルの後に実行」と書くだけで、Claude が文脈を読んで連携します。

```markdown
# X投稿文スキル
description: 動画完成後、SNS展開を依頼されたときに実行する
```

向いている業務：
- 動画完成 → X投稿文 → note記事 → YouTube概要欄
- 文字起こし → 要約 → サムネ用キャッチコピー

## 7. エージェントを作る — AIチーム化（旧 第3回）

### 7-1. エージェントとは

**エージェント＝ ゴールを与えると、自律的に計画→実行→確認→修正を繰り返す別AI**

```
通常のAI:
人間 → 質問 → AI → 回答 → 人間（また質問）...

エージェント:
人間 → ゴール → AI（計画→実行→確認→修正）→ 完了報告
```

### 7-2. なぜ「1人の万能AI」より「専門家チーム」が強いのか

| | 1人の万能AI | 専門家チーム |
| --- | --- | --- |
| コンテキスト | 情報が混ざって汚染 | 各自独立・クリーン |
| 精度 | 役割が多いと落ちる | 専門特化で高精度 |
| コスト | 常にフルモデル | 安いモデルを使い分け |
| セキュリティ | 全ツールにアクセス可 | 必要な権限だけ渡せる |

### 7-3. スキル vs エージェント

| | スキル（skills/） | エージェント（agents/） |
| --- | --- | --- |
| 動かす人 | メインClaude が自分で実行 | メインClaude が **別のAI** に委譲 |
| コンテキスト | メインと共有 | 完全独立（別ウィンドウ） |
| 汚染リスク | あり | なし |
| ツール権限 | 全部 | 最小限に絞れる |
| モデル | 固定 | 安いモデルに変えられる |

> 💡 **使い分けの目安**：定型作業なら skills、独立作業＆コンテキスト分離したいなら agents。

### 7-4. エージェントファイルの書き方

```
.claude/agents/
├── researcher.md    ← 情報収集（Web検索・読み取り）
├── writer.md        ← 執筆（記事・投稿文）
└── reviewer.md      ← チェック（ファクト確認・品質保証）
```

```markdown
---
name: researcher
description: Web検索・情報収集が必要なときに自動委譲される。要約・分析はしない。
tools: WebSearch, WebFetch, Read
model: <軽量モデルを指定（例：Haiku系）。最新名・利用可否は公式に要確認>
---

## 役割
情報収集に特化。要約・判断はせず、収集した情報をそのまま返す。
```

### 7-5. 委譲の2パターン

**パターンA：逐次委譲（直列チェーン）**

```
ユーザー: 「競合調査をまとめてレポート化して」
  ↓
メイン → researcher（調査）
  ↓
メイン → writer（執筆）
  ↓
メイン → reviewer（チェック）→ 完成物
```

向いている業務：議事録作成、提案書、動画→記事化など工程に順番があるもの

**パターンB：Fork-Join 並列**

```
ユーザー: 「3市場の調査をまとめて」
  ↓
メイン: 3体同時起動（Fork）
  researcher_A / researcher_B / researcher_C
  ↓
全員完了後にメインが統合（Join）
```

向いている業務：複数候補比較、複数チャンネル分析、複数記事一括校正など

> 💡 タスクを **小さく分解できればできるほど**、並列の効果が増します。

## 8. HITL — 人間が止めるべき場所を設計する

### 8-1. HITL（Human in the Loop）とは

AI が自動で動く流れの中に、**人間が確認するタイミングを意図的に組み込む** こと。

- 全自動 → リスク大
- 全手動 → 自動化の意味なし
- **その中間を設計する** のが HITL

### 8-2. いつ入れるか — 黄金律

> **「取り消せないか？ 自分以外に影響が及ぶか？」 → YESなら確認を入れる**

| | 取り消せる | 取り消せない |
| --- | --- | --- |
| 影響小 | 自動でOK | 確認推奨 |
| 影響大 | 確認推奨 | **必ず確認（MUST）** |

### 8-3. どこに書くか — 「操作に最も近いファイル」に書く

| ファイル | 何を書くか |
| --- | --- |
| `CLAUDE.md` | コア原則 1行のみ（200行以内維持） |
| 各スキル | そのスキル固有のHITL |
| 各エージェント | そのエージェント固有のHITL |

```markdown
# CLAUDE.md（1行で十分）
- 取り消せない操作は必ず確認してから実行すること
```

```markdown
# agents/mailer.md（送信エージェント）
description: メール・Slack送信専門

## HITLルール
- 送信前に必ず内容を表示して「送信してよいですか？」と確認すること
- 絶対に確認なしで実行しないこと
```

```markdown
# agents/researcher.md（読み取りのみ）
description: Web検索・情報収集専門。読み取りのみで取り消せる操作なし。
# → HITL不要
```

### 8-4. シーン別の入れ方

| 操作 | HITLの入れ方 |
| --- | --- |
| Slack/メール送信 | 送信前に内容を表示して承認 |
| Notion更新 | 差分を表示して「更新してよいですか？」 |
| ファイル削除 | 削除対象リストを提示して確認 |
| 5段階以上の長いタスク | 3段階目で中間確認 |
| 外部API呼び出し | エンドポイント・パラメータを表示して確認 |
| **動画書き出し（上書き）** | **書き出し先パスを必ず表示して確認** |

> 「待て」「検証しろ」の2ルールを徹底するだけで、やり直し率が体感で大幅に減ります。

## 9. コストとモデル使い分け

Claude Code は **使うほどコストがかかります** が、設計次第で大幅に削減できます。

### 9-1. モデル選び — 相対的な使い分け

Claude には用途別に **複数のモデル階層** があり、それぞれ精度・速度・価格が違います。
**具体的なモデル名・価格・Claude Code のデフォルトは時期によって変わる** ため、
実際の数値は必ず公式ページで確認してください：

- 公式 Pricing: <https://www.anthropic.com/pricing>
- Claude Code 公式ドキュメント: <https://docs.anthropic.com/en/docs/claude-code>

| 階層 | 役割の目安 | 向いている業務 |
| --- | --- | --- |
| **上位（最高精度）** | 重要な意思決定・複雑推論・自己検証 | 設計・影響大な文章 |
| **中位（汎用）** | ハイブリッド推論・自律実行・執筆 | 議事録・記事・コード・エージェントのメイン |
| **軽量（高速・安価）** | 大量・単純タスク | 情報収集・データ変換・整形・Sub-agent |

> 💡 **使い分けの考え方**：軽い調査や整形は安価なモデルに委譲、重要判断だけ上位モデルに任せる。

### 9-2. コスト削減3つのワザ（一般原則）

1. **プロンプトキャッシュ** — `CLAUDE.md` 等の不変部分をキャッシュして再利用
2. **バッチ処理** — リアルタイム性不要なタスクをまとめて処理
3. **軽量モデルへの委譲** — Sub-agent の単純作業を安価なモデルに切り出す

> 各削減率は条件依存です。最新の仕様・割引率は公式ドキュメントで確認してください。

### 9-3. 設計のピラミッド

```
      上位モデル（頭取）       ← ここぞの判断だけ
        ／    ＼
      中位モデル（指揮官）     ← デフォルト・全体統括
       ／    ＼
   軽量モデル × 複数（実働）   ← 並列の単純作業
```

### 9-4. 使用量を見るコマンド

```
/cost
```

何に一番使っているかを把握する。Sub-agent化や軽量モデル移行の候補を1つでも見つけたら勝ち。

## 10. 全体設計の最終形

3回ぶんの学びを1つのフォルダ構成に統合すると、こうなります。

```
your-project/
├── CLAUDE.md                ← 【基礎】コア原則のみ（200行以内）
│                              「取り消せない操作は必ず確認」の1行から始める
├── .claude/
│   ├── settings.json        ← 【基礎】ツール権限の allow/deny
│   │                          deny: rm -rf *、.env 読み取り
│   ├── commands/            ← /xxx で明示的に呼ぶ独自コマンド
│   │
│   ├── skills/              ← 【スキル編】自律発動ワークフロー
│   │   └── 字幕生成/
│   │       ├── SKILL.md         ← 概要・トリガー・基本手順
│   │       │                      HITLルールはここに（書き出し前に確認）
│   │       ├── references/      ← 詳細フォーマット・過去事例
│   │       └── data/log.json    ← 実行ログ（永続化）
│   │
│   └── agents/              ← 【エージェント編】専門分業
│       ├── researcher.md     ← HITL不要（読み取りのみ）
│       ├── writer.md         ← HITL不要（ドラフト作成）
│       └── mailer.md         ← HITL必須！（送信＝取り消せない）
```

| ファイル | 書く内容 | 出典 |
| --- | --- | --- |
| `CLAUDE.md` | コア原則のみ・200行以内 | 第1回 |
| `settings.json` | ツール権限の allow/deny | 第1回 |
| `skills/各スキル` | 概要 + Gotchas + HITL + 参照 | 第2回 |
| `agents/各エージェント` | 役割 + モデル + 権限 + HITL | 第3回 |
